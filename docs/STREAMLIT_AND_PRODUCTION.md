# Streamlit и продакшен

План поэтапного вывода монолитного [`packaging_viewer.py`](../packaging_viewer.py) из основного контура.

## Текущее разделение

- **Streamlit** — полный UI и бизнес-логика как сегодня.
- **FastAPI** ([`api/main.py`](../api/main.py)) — read-only каталог; может работать с SQLite (`PACKAGING_DB_PATH`) или PostgreSQL (`PACKAGING_DATABASE_URL`).
- **SPA** ([`web/`](../web/)) — таблица каталога поверх API.

## Этапы

1. **Параллельный запуск:** в prod поднять API + (по желанию) статическую сборку SPA за reverse proxy; Streamlit оставить для сотрудников по внутреннему URL.
2. **Перенос сценариев:** по одному выносить экраны (печать, cutii, планировщик) в API + SPA; Streamlit вызывать только для ещё не перенесённого.
3. **Сужение Streamlit:** огранить доступ (VPN, Basic auth, отдельный порт), отключить публичный ingress.
4. **Отключение в prod:** когда паритет функций достигнут, оставить Streamlit только локально для админ-задач или удалить.

## Переменные окружения (ориентир)

| Сервис    | Переменные |
|-----------|------------|
| API       | `PACKAGING_DB_PATH` или `PACKAGING_DATABASE_URL` |
| Streamlit | как сейчас (путь к БД в коде/секретах) |
| SPA build | `VITE_API_URL` — базовый URL API, если без прокси |

## Риски

- Две запись в одну SQLite с API и Streamlit одновременно — только если позже добавить write в API; до тех пор SQLite read-only с API безопасен параллельно со Streamlit.
- После перехода на Postgres — единая БД для API и фоновых воркеров; Streamlit перевести на тот же DSN или оставить на SQLite до полного отказа.
